# Git 操作の安全ルール

## 基本原則

- **AIエージェントは git add / commit / push をユーザーが明示的に依頼した場合のみ実行すること。**
- **ユーザーへの確認・説明はすべて日本語で行うこと。**

## ルール一覧

### 1. 自発的な提案の禁止

- **コード変更後はユーザーの指示を待つこと**
- **プッシュはユーザーから明示的に指示があった場合のみ実行すること**
- 変更が完了しても、git 操作はユーザーの指示を待つ

### 2. ユーザー依頼時の実行範囲

ユーザーが以下のように依頼した場合、**ステージング（git add）も含めて** 一連の操作として実行する:

| ユーザーの依頼 | 実行する操作 |
|---|---|
| 「コミットして」 | git add → git commit |
| 「プッシュして」 | git push |
| 「コミットプッシュ」「コミットプッシュして」 | git add → git commit → git push |

### 3. 範囲が不明な場合は必ず確認

どのファイルをステージングすべきか判断できない場合は、**必ず確認を取ること**。
**git add する前に、ステージング対象を必ずユーザーに確認すること。**

確認の例:
- 「以下のファイルをコミット対象にしてよいですか？」
- 「変更が複数ありますが、すべてコミットしますか？それとも一部だけですか？」

### 4. ブランチの整合性チェック

コミット前に以下を確認し、問題があれば**確認を取ること**。

#### コミット内容とブランチ名の不一致

ブランチ名から推測される目的と、コミット内容が明らかに異なる場合は確認する。

例:
- ブランチ `feature/login` なのにログインと無関係な修正が含まれている
- ブランチ `fix/header-bug` なのに新機能のコードが含まれている

確認の例: 「ブランチ名は `feature/login` ですが、ログインと関係のない変更が含まれています。このままコミットしてよいですか？」

#### 保護ブランチでの直接作業

`main`、`master`、`develop` などのブランチで直接作業している場合、**初回のみ** 確認する。

確認の例: 「現在 `main` ブランチで直接作業しています。作業用ブランチに切り替えますか？」

ユーザーが「そのままでいい」と回答した場合、**同一セッション内では2回目以降は聞かない**。

> [!WARNING]
> Bitbucket 環境では git-branch-bitbucket のルールが適用される。保護ブランチ（main/master/develop）への直接プッシュは git-branch-bitbucket §4 を参照すること。

### 5. Worktree による作業分離

現在のブランチの目的と異なる修正・作業が発生した場合、**git worktree を使って別ディレクトリで作業する**。

#### 適用条件

以下のいずれかに該当する場合、worktree での作業分離を提案する:

- ユーザーから現在のブランチの目的と異なる修正依頼を受けた場合
- 現在のブランチにステージング済みまたはコミット済みの変更がある状態で、別ブランチの作業が必要になった場合

#### 手順

1. **状況を説明し、worktree の使用を提案する**:

   > [!WARNING]
   > 現在 `<current-branch>` で作業中ですが、依頼された内容は別ブランチ（`<target-branch>`）での作業が適切です。git worktree で別ディレクトリに作業環境を作成してよいですか？

2. **適切なブランチを特定する**:
   - 既存ブランチで適切なものがあればそれを使用する
   - なければユーザーに確認の上、新規ブランチを作成する

3. **ユーザーの承認を得てから worktree を作成する**:

   ```powershell
   git worktree add <パス> <ブランチ>
   ```

4. **worktree ディレクトリで作業を行う**

5. **作業完了後、必ず worktree を削除する**:

   ```powershell
   git worktree remove <パス>
   ```

   > [!WARNING]
   > worktree の削除忘れはリポジトリの整合性に影響するため、**作業完了時に必ず実行する**。ユーザーから完了の合図があった時点で速やかに remove すること。

#### worktree の整合性維持（prune）

worktree のディレクトリが手動削除などで存在しなくなっている場合、**状況を説明してから `git worktree prune` を実行する**。

検知タイミング:
- `git worktree list` の実行時に、存在しないディレクトリが表示された場合
- worktree ディレクトリにアクセスしようとして存在しなかった場合

対応手順:

1. **状況を説明する**:

   > [!NOTE]
   > worktree `<パス>` のディレクトリが存在しません。手動で削除された可能性があります。`git worktree prune` を実行して worktree の管理情報を整理します。

2. **prune を実行する**:

   ```powershell
   git worktree prune
   ```

3. **結果を確認する**:

   ```powershell
   git worktree list
   ```

#### 注意事項

- **worktree の作成・削除は必ずユーザーの承認を得てから実行すること**
- worktree 内での作業中も、コミット・プッシュに関する他のルール（§1〜§4）はすべて適用される
- 同一ブランチを複数の worktree で同時にチェックアウトすることはできない

### 6. コミットメッセージ

**Conventional Commits 形式** + **JEAD コミットメッセージルール** に従う。

#### フォーマット

```
<type>(<scope>): <description> <課題番号>

[本文（任意）]

[フッター（任意）]
Co-authored-by: Copilot <223556219+Copilot@users.noreply.github.com>
```

#### type 一覧

| type | 用途 |
|:---|:---|
| `feat` | 新機能の追加 |
| `fix` | バグ修正 |
| `docs` | ドキュメントのみの変更 |
| `style` | コードの意味に影響しない変更（空白、フォーマット等） |
| `refactor` | バグ修正でも機能追加でもないコード変更 |
| `test` | テストの追加・修正 |
| `chore` | ビルドプロセスや補助ツールの変更 |

#### 記述ルール

- **description は「〜の追加」「〜の修正」の体言止め形式で書くこと**
- **コミットメッセージの description は句読点なしの体言止め形式で書くこと**
- `scope` は変更対象を簡潔に示す（省略可）
- **Jira 課題番号** がある場合は description の末尾にそのまま付ける（例: `HRSJP-46`）
- 課題番号が不明な場合はユーザーに確認する（なければ省略可）
- `Co-authored-by` トレーラーを必ず付ける

#### サンプル

```
feat(cart): カートに数量変更機能を追加 HRSJP-123

fix(auth): トークン期限切れのリダイレクトループを修正 HRSJP-456

docs(readme): セットアップ手順を更新 HRSJP-111

chore: .gitignore にビルド成果物を追加
```

```markdown
<!-- ❌ Bad: 体言止めではない -->
feat(cart): カートに数量変更機能を追加します HRSJP-123

<!-- ❌ Bad: 句読点がある -->
fix(auth): リダイレクトループを修正。 HRSJP-456

<!-- ❌ Bad: 修正内容が不明 -->
fix: 不具合修正 HRSJP-456
```

# Markdown ドキュメント作成スキル

## ドキュメント種別の判定

ユーザーの依頼内容から、作成するドキュメントの種別を判定する。
種別ごとにテンプレートと注意点が異なるため、適切なリファレンスを参照してから書き始める。

| 種別 | 典型的な依頼 | 参照先 |
|---|---|---|
| README | 「README を書いて」「プロジェクト説明を書いて」 | `references/readme-guide.md` |
| 議事録 | 「議事録を作って」「会議メモをまとめて」 | `references/meeting-minutes.md` |
| 調査・分析レポート | 「調査結果をまとめて」「分析レポートを書いて」「ログを分析して」 | `references/report-guide.md` |
| 比較資料 | 「○○と○○を比較して」「料金比較を作って」 | `references/comparison-guide.md` |
| 計画書 | 「移行計画を書いて」「プロジェクト計画を作って」 | `references/plan-guide.md` |
| ヒアリング・説明資料 | 「説明資料を作って」「ヒアリング資料を書いて」「提案書を作って」 | `references/presentation-guide.md` |
| WBS・ガントチャート | 「WBS を作って」「スケジュールを Markdown で」 | `references/wbs-gantt.md` |
| ロードマップ | 「ロードマップを作って」「マイルストーンをまとめて」 | `references/roadmap-guide.md` |
| メモ・ノート | 「メモを書いて」 | `references/memo.md` |
| 図解 | 図・チャート・ダイアグラムの作成依頼全般 | `references/diagram-patterns.md` |
| その他・不明 | 上記に当てはまらない | 共通ルールのみ適用 |

> [!TIP]
> 種別が不明な場合はユーザーに確認する。確認しなくても明らかな場合（例:「README.md を書いて」）はそのまま進めてよい。

## 共通ルール

すべてのドキュメント種別に適用する基本ルール。

### 文書構造

- 見出しは `#` → `##` → `###` の順で使い、レベルを飛ばさない
- 4階層（`####`）以上は避ける — 構造が深すぎる場合はセクション分割を検討する
- リストは `-` を使用する（`*` は使わない）
- 日本語で記述する（ユーザーの指示に従う）

### コードブロック

コードブロックが必要な場面では、必ず言語名を指定する。

```markdown
```python
print("Hello")
```
```

ただし議事録やメモなど、コードが登場しないドキュメントでは無理にコードブロックを使わない。

### テーブル

- `|---|` 形式で整形する
- 列が 4 列以上になる場合はリストへの変換を検討する
- 数値列は右寄せ（`---:`）、見出し列は中央寄せ（`:---:`）

## GitHub Markdown Alerts

補足・注意事項には **GitHub Markdown Alerts** を使用する。
プレーンテキストの「注意:」「ポイント:」「※」は使わない。

### 5 種類のアラートと使い分け

```markdown
> [!NOTE]
> 補足情報。知っておくと便利な追加情報。

> [!TIP]
> ヒント・コツ。より良いやり方やおすすめの方法。

> [!IMPORTANT]
> 重要な情報。理解しておくべき要点やポイント。

> [!WARNING]
> 注意。問題が起きる可能性がある内容。

> [!CAUTION]
> 危険・リスク。データ損失や破壊的操作に関する警告。
```

| 状況 | 使うアラート |
|---|---|
| 「ちなみに」「補足ですが」 | `[!NOTE]` |
| 「おすすめ」「こうするとよい」 | `[!TIP]` |
| 「ポイント」「要点」「必ず確認」 | `[!IMPORTANT]` |
| 「注意」「〜しないでください」 | `[!WARNING]` |
| 「データが消える」「元に戻せない」 | `[!CAUTION]` |

## Mermaid 図解の活用

文章だけでは伝わりにくい**構造・流れ・関係性**がある内容には、ユーザーから依頼されなくても Mermaid 図解を積極的に挿入する。
GitHub の Markdown は ` ```mermaid ` コードブロックをネイティブでレンダリングするため、画像の生成やツールのインストールは不要。

図の種類の選び方や書き方は `references/mermaid-examples.md` を参照する。

> [!IMPORTANT]
> **スケジュール・ガントチャートには Mermaid `gantt` を使わない。** WBS（`type: wbs`）やロードマップ（`type: roadmap`）のスケジュールはテーブル形式で記述する。HTML 変換時に CSS ガントチャートが自動生成されるため、Mermaid は不要。フローチャート・シーケンス図・状態遷移図など他の Mermaid 図は引き続き使用可能。

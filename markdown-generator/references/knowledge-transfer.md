# ナレッジトランスファー（KT）記録ガイド

## HTML 変換を前提とした frontmatter

html-generator スキルで HTML に変換する場合は、YAML frontmatter に `type: knowledge-transfer` を付与する。

```yaml
---
type: knowledge-transfer
title: KTタイトル
date: YYYY-MM-DD
---
```

## KT テンプレート

```markdown
# [KTタイトル]

- **日時**: YYYY-MM-DD HH:MM〜HH:MM
- **場所**: [会議室名 / オンライン（Teams 等）]
- **伝達者**: [名前（役割・担当領域）]
- **受領者**: [名前1、名前2]
- **記録者**: [名前]

## 対象範囲

[今回のKTで扱う範囲と目的を簡潔に記述]

## 内容

### 1. [トピック1]

[伝達された内容を文章で記述。業務知識であれば背景・経緯・判断根拠を含める]

### 2. [トピック2]

[手順の引き継ぎであれば番号付きリストで記述してよい]

## 確認事項

| No. | 事項 | 状態 | 備考 |
|---:|---|---|---|
| 1 | [受領者が理解できた点・残った疑問] | 理解済 / 要確認 | [補足] |

## フォローアップ

- [ ] [追加確認が必要な事項] — 担当: [名前] / 期限: YYYY-MM-DD
- [ ] [次回KTで扱う予定の事項] — 担当: [名前] / 期限: YYYY-MM-DD
```

## セクションの解説

### 対象範囲

今回のKTで何を扱い、何を扱わないかを明示する。受領者が後から読み返したときに、このKTのスコープがすぐにわかるようにする。

### 内容

KTの本体。トピックごとにセクションを分け、内容の性質に応じて書き方を変える。

| 内容の性質 | 書き方 |
|---|---|
| 業務知識・背景・経緯 | 文章で記述。なぜそうなっているかの判断根拠を含める |
| 手順・操作方法 | 番号付きリスト（`1.` `2.` `3.`）で記述してよい |
| 技術知識・設計思想 | 文章で記述。必要に応じて Mermaid 図解を挿入 |
| 人・組織の関係性 | 文章で記述。問い合わせ先やエスカレーションパスを明記 |
| 共有された数値・データ | テーブルで記述 |

### 確認事項

受領者の理解状態を記録する。KT完了後に「何が伝わり、何がまだ曖昧か」を可視化するためのセクション。

- No. 列は右寄せ（`---:`）
- 状態は「理解済」「要確認」のいずれかを記入する

### フォローアップ

- チェックボックス（`- [ ]`）形式を使う
- 各項目に担当者と期限を併記する

## KT 固有のルール

### 書式

- 文体はだ・である調、体言止めを交えて簡潔に書く
- トランスクリプトに存在しない情報を加筆しない

### 記述のポイント

- 伝達者の発言は詳細かつ網羅的に記録する。業務知識・背景・経緯の説明では箇条書きを使わず文章としてまとめる
- 手順・操作方法の説明は番号付きリストで記述してよい
- 発言者の引用が必要な場合は `> [名前]: 発言内容` の形式を使う
- 暗黙知や口頭でしか伝わらない情報（判断基準、例外対応、過去の失敗事例など）を重点的に記録する

### 品質基準

- ビジネス文書としてチーム内共有可能なレベルを目指す。手作業修正が全体の 2 割以内に収まることが目安
- 見出しの粒度・順序は伝達内容の流れと重要性に応じて最適化してよい（テンプレートの構成に固執しない）

# ヒアリング・説明資料作成ガイド

ヒアリング冒頭の説明資料、提案資料、社内説明資料など、「相手に説明して理解・合意を得る」ためのドキュメント。
プレゼンのスライド的な構成で、1セクション1テーマに絞り、視覚的にわかりやすくする。

## HTML 変換を前提とした構造規約

この説明資料は md-to-html スキルで HTML に変換される場合がある。
HTML 変換時にページ切り替えレイアウトが適用されるよう、以下の規約に従う。

### frontmatter

必ず YAML frontmatter を付与する。`layout` で HTML のナビゲーション形式を指定する。

```yaml
---
type: presentation
layout: slide
title: 資料タイトル
target: 対象者
date: YYYY-MM-DD
---
```

### layout の種類

| layout | ナビゲーション | 適した場面 |
|---|---|---|
| `slide` | 左右矢印キーで横スライド遷移 | ヒアリング資料、順序立てた説明 |
| `spa` | カード/リンクをクリックして各ページに遷移 | テーマ別の解説、トピック選択型の資料 |

### ページ区切りのルール

- `##`（h2 見出し）がページ区切りになる
- `---`（水平線）もページ区切りとして使える
- 1ページ = 1テーマを原則とする

### 構造テンプレート（slide レイアウト）

```markdown
---
type: presentation
layout: slide
title: ○○ヒアリング資料
target: ○○様
date: 2026-03-08
---

## 本日のアジェンダ

1. 背景と目的
2. 現状の整理
3. 提案内容
4. スケジュール
5. 確認事項

---

## 1. 背景と目的

### 背景

現状の課題や経緯を簡潔に。

### 目的

この説明で伝えたいこと・合意したいこと。

---

## 2. 現状の整理

現在の状態を図やテーブルで整理。

---

## 3. 提案内容

### 概要

提案の要点を 1〜3 文で。

> [!IMPORTANT]
> 特に確認・合意いただきたいポイント。

---

## 4. スケジュール

| フェーズ | 期間 | 内容 |
|---|---|---|
| Phase 1 | MM月〜 | ○○ |
| Phase 2 | MM月〜 | ○○ |

---

## 5. 確認事項

- [ ] 確認事項1
- [ ] 確認事項2
```

### 構造テンプレート（spa レイアウト）

```markdown
---
type: presentation
layout: spa
title: AI コーディングツールの進化
date: 2026-03-08
---

## 概要

全体の導入文。各トピックをカードとして一覧表示し、クリックで詳細ページへ遷移する。

---

## 第1世代: コード補完

### 特徴

- 行レベルの補完
- IDE プラグイン型

### 代表ツール

| ツール名 | 時期 | 特徴 |
|---|---|---|
| GitHub Copilot | 2021年〜 | 行・関数レベルの補完 |

---

## 第2世代: IDE統合チャット

### 特徴

- エディタ内でチャット
- コンテキスト理解

### 代表ツール

| ツール名 | 時期 | 特徴 |
|---|---|---|
| Cursor | 2024年〜 | AI ネイティブ IDE |

## まとめ

全体を振り返る結論ページ。
```

> [!NOTE]
> `spa` レイアウトでは最初のページ（概要）がトップページとなり、各 `##` セクションがカードとして一覧表示される。
> カードをクリックすると該当セクションの詳細ページに遷移する。

## 説明資料の種別ごとの補足

### ヒアリング資料

相手から情報を引き出すための資料。

- **アジェンダを冒頭に置く**: 話の流れを先に共有して安心感を持たせる
- **確認事項をチェックリスト化**: ヒアリングで確認したい項目を `- [ ]` で列挙する
- **現状の理解を先に示す**: 「こう理解していますが合っていますか」の形で提示

```markdown
## 現状の理解

> [!NOTE]
> 以下は事前情報をもとにした現状理解です。認識に相違があればご指摘ください。

| 項目 | 現状の理解 |
|---|---|
| 対象システム | ○○ |
| 利用ユーザー | 約○名 |
| 課題 | ○○が○○になっている |
```

### 提案・説明資料

こちらの考えを伝えて合意を得るための資料。

- **1セクション1メッセージ**: 各セクションで伝えたいことを1つに絞る
- **Before / After を明示**: 変更の効果を対比で示す

```markdown
## 導入効果

| 観点 | 現状（Before） | 導入後（After） |
|---|---|---|
| 処理時間 | 約30分 | 約5分 |
| 手動作業 | 毎日3件 | 自動化 |
```

### ダッシュボード・UI イメージ

画面イメージやダッシュボードの構成を説明する資料。

- **画面構成をテキストで示す**: 各要素の配置と目的を説明する
- **データ項目をテーブルで整理**: 表示するデータの定義を明確にする

```markdown
## ダッシュボード構成

| エリア | 表示内容 | データソース |
|---|---|---|
| ヘッダー | KPI サマリー | ○○テーブル |
| メイン | トレンドグラフ | 月次集計 |
| サイドバー | フィルター | マスタ |
```

## 図解の活用

説明資料では現状と提案の対比や、全体像の可視化が特に有効。図の種類の選び方は `diagram-patterns.md` を参照する。

| 場面 | 推奨する図 |
|---|---|
| 全体像・アーキテクチャ | `flowchart` + `subgraph` |
| 作業やユーザーの流れ | `flowchart LR` |
| Before / After の対比 | `flowchart` を2つ並べる、または `subgraph` で対比 |
| 関係者間のやりとり | `sequenceDiagram` |
| スケジュール | テーブル形式（HTML 変換時に CSS ガントチャートが自動生成） |

## 説明資料固有のルール

- **セクション間に `---`（水平線）を入れる** — スライドの区切りのように視覚的に分離する
- **1セクション1テーマ** — 1つのセクションに複数のテーマを詰め込まない
- **アジェンダを冒頭に置く** — 全体の流れを先に示す
- **確認事項・質疑をラストに置く** — 合意すべきポイントを最後にまとめる
- **図やテーブルを積極的に使う** — 文章の壁を避け、視覚的に伝える
- **対象者と日付を明記する** — 誰向けの資料か、いつの情報かを示す

# README.md 作成ガイド

## 最初にやること

1. **コミット先が GitHub か Bitbucket か確認する**（同一セッションで 1 回のみ）
2. Bitbucket の場合 → 下記「Bitbucket の制約」を適用
3. GitHub の場合 → 全記法が利用可能

### Git サービスの確認

コミット先の Git サービスによって使える記法が異なる。
**セッション内でまだ確認していない場合**、以下を日本語でユーザーに確認する（同一セッションでは 1 回のみ）。

- GitHub か Bitbucket（Atlassian）か、その他か

### Bitbucket の制約

Bitbucket の README.md では以下の記法が**使用不可**である。

- Mermaid 図（` ```mermaid ` コードブロック）
- GitHub Markdown Alerts（`> [!NOTE]` 等）
- 脚注（`[^1]`）
- 折りたたみ（`<details>` / `<summary>`）
- 数式（LaTeX `$...$` / `$$...$$`）

Bitbucket の場合、これらの代わりに以下を使う。

| GitHub 記法 | Bitbucket 代替 |
|---|---|
| Mermaid 図 | テキストで説明、または画像を添付 |
| Alerts | `**注意:**` や `**ポイント:**` のように太字で表現 |
| 折りたたみ | 見出しでセクション分け |
| 数式 | プレーンテキストで記述 |

## README.md 構成テンプレート

README.md を新規作成する場合の推奨構成:

```markdown
# プロジェクト名

概要を 1〜2 文で。

## 前提条件

- 必要なツール・バージョン

## セットアップ

手順をコードブロック付きで。

## 使い方

基本的な使用例。

> [!TIP]
> 便利な使い方やショートカット。

> [!IMPORTANT]
> 守るべきルールや制約。
```

### セクションの取捨選択

プロジェクトの規模や性質に応じて、以下のセクションを追加・省略する。

| セクション | いつ必要か |
|---|---|
| 前提条件 | 環境構築が必要な場合 |
| セットアップ | インストール手順がある場合 |
| 使い方 | ほぼ常に必要 |
| 設定 | 環境変数や設定ファイルがある場合 |
| トラブルシューティング | よくあるエラーがある場合 |
| コントリビューション | OSS やチーム開発の場合 |
| ライセンス | OSS の場合 |

## 図解の活用

README は「プロジェクトの全体像を素早く伝える」ドキュメントであるため、文章だけでは伝わりにくい構造・流れがあれば Mermaid 図解を積極的に挿入する。
図の種類の選び方や書き方は `diagram-patterns.md` と `mermaid-examples.md` を参照する。

### README でよく使う図の種類

| 場面 | 推奨する図 | Mermaid 宣言 |
|---|---|---|
| システム全体の構成 | アーキテクチャ図 | `flowchart` + `subgraph` |
| 処理や操作の流れ | フローチャート | `flowchart LR` / `TD` |
| API やサービス間の通信 | シーケンス図 | `sequenceDiagram` |
| データの関連 | ER 図 | `erDiagram` |
| Git ブランチ運用 | Gitgraph | `gitGraph` |

### 図を入れる位置

- **概要の直後**: プロジェクト全体のアーキテクチャ図を置くと、読み手が構造を即座に把握できる
- **セットアップセクション内**: 環境構成が複雑な場合、構成図を手順の前に置く
- **使い方セクション内**: ユーザー操作のフローが複雑な場合

> [!TIP]
> README の図は概要レベルに留める。詳細な設計図は別のドキュメントに分離し、README からリンクする。

## README.md 固有のルール

- 概要は冒頭に置き、1〜2 文で完結させる — 詳細は後続セクションに書く
- セットアップ手順には必ず**コードブロック**を含める（コピペ可能にする）
- スクリーンショットよりも Mermaid 図やテキストを優先する — テキストベースの図は検索・更新・差分確認が容易なため

# ロードマップ・マイルストーン作成ガイド

プロダクトやプロジェクトの中長期的な方向性を示すドキュメント。
「いつ・何を達成するか」をフェーズとマイルストーンで俯瞰的に伝える。
WBS が個々のタスクを管理するのに対し、ロードマップは戦略レベルの全体像を共有するために使う。

## HTML 変換を前提とした frontmatter

md-to-html スキルで HTML に変換する場合は、YAML frontmatter に `type: roadmap` を付与する。
`layout` フィールドでビジュアルパターンを指定できる。

```yaml
---
type: roadmap
layout: gantt          # gantt / timeline / swimlane / feature / chevron
title: ロードマップタイトル
project: プロジェクト名
date: YYYY-MM-DD
---
```

### レイアウトパターン

| layout | 特徴 | 適するケース |
|---|---|---|
| `gantt`（デフォルト） | 月単位 CSS ガントチャート + フェーズグループ + TODAY マーカー | プロジェクト全体のスケジュール管理 |
| `timeline` | 水平タイムライン + マイルストーンマーカー + KPI ボックス | 経営層向け・DX 推進など戦略レベルの概要 |
| `swimlane` | カテゴリ別スイムレーン × 四半期グリッド | 複数チーム・領域の並行作業を俯瞰 |
| `feature` | シェブロンフェーズ + 製品別スイムレーン + 機能カード | プロダクトの機能リリース計画 |
| `chevron` | 5 フェーズシェブロン + 交互アノテーション | シンプルなフェーズ概要・役員説明向け |

- `layout` を省略した場合、HTML 変換時にパターン選択を求められる（スケルトン HTML をブラウザでプレビューして選べる）
- 迷ったら `gantt` を選ぶ

## 構成テンプレート

```markdown
---
type: roadmap
layout: gantt
title: ロードマップタイトル
project: プロジェクト名
date: YYYY-MM-DD
---

# [プロジェクト名] ロードマップ

## ビジョン

このロードマップが目指すゴールと背景を簡潔に記述する。

## 現在のステータス

| 項目 | 内容 |
|---|---|
| 現在のフェーズ | Phase 2: 開発 |
| 次のマイルストーン | ベータリリース（MM/DD） |
| 全体進捗 | 40% |

## タイムライン

フェーズとマイルストーンの全体像を表形式で定義する。

| フェーズ | タスク | 開始 | 終了 | 状態 |
|---|---|---|---|---|
| Phase 1: 企画 | 市場調査・要件定義 | 2026-01-06 | 2026-02-28 | 完了 |
| Phase 1: 企画 | 企画承認（MS） | 2026-02-28 | — | 完了 |
| Phase 2: 開発 | 基本機能開発 | 2026-03-01 | 2026-05-31 | 進行中 |
| Phase 2: 開発 | ベータリリース（MS） | 2026-05-31 | — | 注意 |
| Phase 3: 検証 | ベータテスト・改善 | 2026-06-01 | 2026-07-31 | 未着手 |
| Phase 3: 検証 | 品質基準クリア（MS） | 2026-07-31 | — | 未着手 |
| Phase 4: リリース | GA 準備 | 2026-08-01 | 2026-08-31 | 未着手 |
| Phase 4: リリース | 正式リリース（MS） | 2026-08-31 | — | 未着手 |

> **タイムライン表の書き方:**
> - マイルストーンは末尾に `（MS）` を付けて区別する
> - 状態: `完了` / `進行中` / `注意` / `未着手` / `遅延`
> - HTML 変換時にこの表が選択した layout のビジュアル表現に変換される

| リスク | 影響 | 対策 |
|---|---|---|
| ○○の遅延 | Phase 3 開始が後ろ倒し | 並行作業で吸収 |

## 変更履歴

| 日付 | 変更内容 | 変更者 |
```

## フェーズの書き方

各フェーズは以下の構造で書く:

1. **`## Phase N: フェーズ名（期間）`** — 見出しに番号・名前・期間を含める
2. **`### 目標`** — フェーズのゴールを簡潔に
3. **`### 主な成果物`** — リスト形式で成果物を列挙
4. **`### マイルストーン: 名前 — 日付`** — フェーズの完了判定ポイント

## マイルストーンの書き方

マイルストーンはフェーズの区切りとなる重要な達成点。以下を含める:

- **名前と日付** — 見出しに明示
- **完了条件** — 何をもって達成とするか
- **判定基準** — Go/No-Go の基準があれば記載

## タイムライン表でのステータス指定

| 状態 | 意味 | HTML 変換時のスタイル |
|---|---|---|
| 完了 | 達成済みのフェーズ・マイルストーン | 緑系（`.done`） |
| 進行中 | 現在取り組んでいるフェーズ | 橙系（`.active`） |
| 注意 | 遅延リスクや注目すべきマイルストーン | 橙系（`.at-risk`） |
| 未着手 | 今後のフェーズ | グレー系（`.upcoming`） |
| 遅延 | 予定より遅れている | 赤系（`.delayed`） |

## ロードマップ固有のルール

- **フェーズ番号を付ける** — `Phase 1`, `Phase 2` で順序を明確にする
- **マイルストーンには日付を必ず含める** — 期限のないマイルストーンは意味がない
- **現在のフェーズを明示する** — `[!IMPORTANT]` アラートや太字で「今ここ」を示す
- **成果物を具体的に書く** — 「開発を進める」ではなく「API ドキュメント完成」のように
- **変更履歴を残す** — ロードマップは更新されるものなので、いつ何が変わったかを記録する
- **タスク名は短くする** — ビジュアルのバー内に収まるよう簡潔に。詳細はフェーズ詳細セクションに書く

## 遅延・先行時の表現ルール

ロードマップに遅延や日程変更がある場合、以下のパターンで統一的に表現する。

### Markdown 上での書き方

- **日付の変更**: 取消線で旧日付を示し、新日付を続ける — `~~5/31~~ 6/14`
- **遅延の注記**: 括弧で遅延幅を明示 — `（2週間遅延）`
- **期間の変更**: 当初予定を括弧内に残す — `3月15日〜6月14日（当初予定: 3月1日〜5月31日）`
- **影響の説明**: `[!WARNING]` アラートで遅延の影響と対策を記述

### タイムライン表での表現

- 遅延タスクの状態列に `遅延` を指定
- 日程は実績・最新予定に更新する
- タスク名に遅延情報を含めてよい — `企画承認(2週間遅延)（MS）`

### ステータスカード・マイルストーンでの表現

| 状態 | ステータスカード | タイムラインのドット | マイルストーンマーカー |
|---|---|---|---|
| 順調 | `.on-track`（緑枠） | `.done`（緑丸） | `.done`（緑バッジ） |
| 注意 | `.at-risk`（橙枠） | `.active`（橙丸） | `.active`（橙バッジ） |
| 遅延 | `.delayed`（赤枠） | `.delayed`（赤丸） | `.delayed`（赤バッジ） |
| 未着手 | デフォルト（枠なし） | `.upcoming`（灰丸） | `.upcoming`（灰バッジ） |

### 例: 遅延時のフェーズ記述

```markdown
## Phase 2: 開発（3月15日〜6月14日）

> [!WARNING]
> Phase 1 の企画承認が2週間遅延したため、後続フェーズが後ろ倒し。

### 主な成果物

- 基本機能の実装
- API ドキュメント

### マイルストーン: ベータリリース — ~~5月31日~~ 6月14日（2週間遅延）

ベータリリースの完了条件。
```

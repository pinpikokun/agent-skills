# やってはいけないこと

## アラートの乱用

- `**注意:**` や `※` でアラート相当の情報を書くこと → `[!WARNING]` 等を使う
- アラートを連続・大量に使うこと — 連続するとかえって読みづらくなる:
  - 1セクション内のアラートは **最大2つ** まで
  - 同じ種類のアラートが **3つ以上連続** する場合は、通常のリストやテーブルにまとめる
  - トラブルシューティングのような繰り返し構造には、アラートではなく **見出し＋本文** のシンプルな形式を使う

| 場面 | 代わりに使う書式 |
|:---|:---|
| 同じ種類の注意が3つ以上並ぶ | 通常のリスト（`-`） |
| トラブルシューティングの各項目 | `### エラー名` + 本文 |
| 手順の中の補足が多い | インラインの **太字** や括弧書き |

## テーブルの乱用

テーブルは便利だが、横長になるとスクロールが発生し読みにくくなる。

- 列が **4列以上** の場合はテーブルを避ける（リストや定義リスト形式を検討）
- セル内の文章が **長い** 場合もテーブルを避ける
- 代替: リスト、見出し＋本文、折りたたみ

## 見出しレベルの飛ばし

見出しは `#` → `##` → `###` の順に使い、**レベルを飛ばさない**。

```markdown
<!-- NG: # の次に ##### -->
# プロジェクト概要
##### 背景
##### 目的

<!-- OK: レベル順 -->
# プロジェクト概要
## 背景
## 目的
```

## 太字・絵文字・装飾の過剰使用

過剰な装飾はノイズになり、本当に重要な情報が埋もれる。

```markdown
<!-- NG -->
## **超重要！！！**
**必ず** ***絶対に*** **確認** してください！！！

<!-- OK -->
## 重要なお知らせ
必ず確認してください。
```

目安:
- 1セクションで太字は **1〜2箇所** まで
- 絵文字は **見出しの先頭だけ** に使う
- `!!!` や `???` の多用は避ける

## 巨大な1枚ドキュメント

長大なドキュメントはスクロールが多く読みにくい。

- **100行を超えたら** 目次と折りたたみの導入を検討する
- セクション数が多い場合はファイル分割も検討する

## HTML の多用

GitHub Markdown では `style` 属性はほぼ無視されるため、意図通りに表示されない。

- HTML は `<details>`, `<kbd>`, `<img>` など **Markdown で表現できないもの** に限定する
- レイアウトや色の制御に HTML を使わない（可読性が大幅に低下する）

## リンク URL の直貼り

長い URL がそのまま表示されると、読みにくくレイアウトも崩れる。

```markdown
<!-- NG -->
https://docs.google.com/spreadsheets/d/1aBcDeFgHiJkLmNoPqRsTuVwXyZ/edit#gid=0

<!-- OK: テキストリンク -->
[コンセプトシート（Google Sheets）](https://docs.google.com/spreadsheets/d/1aBcD...)
```

## 深すぎるネスト（入れ子）

深いネストは読みにくく、構造を把握しづらい。

- ネストは **3段まで** にとどめる
- それ以上は見出し（`##`, `###`）で分割する

## 画像だけで情報を伝える

スクリーンショットや画像のみでは、検索できず、アクセシビリティも低い。

- 図は原則 **Mermaid 等のテキストベース** で書く
- スクリーンショットを使う場合は **テキストで補足説明** を添える
- 画像の `alt` テキストは必ず記述する

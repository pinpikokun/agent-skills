# 調査・分析レポート作成ガイド

技術調査、データ分析、ログ分析など、何らかの調査を行いその結果を報告するドキュメント。
読み手が「結論」と「根拠」を素早く把握できる構造にすることが最も重要。

## HTML 変換を前提とした frontmatter

md-to-html スキルで HTML に変換する場合は、YAML frontmatter に `type: report` を付与する。
report は md-to-html の既存レイアウト（縦スクロール型）で変換される。

```yaml
---
type: report
title: レポートタイトル
date: YYYY-MM-DD
---
```

## 構成テンプレート

```markdown
---
type: report
title: レポートタイトル
date: YYYY-MM-DD
---

# レポートタイトル

調査日: YYYY年MM月DD日

---

## 背景

なぜこの調査を行ったのか、1〜3 文で簡潔に。

## 結論（エグゼクティブサマリー）

> [!IMPORTANT]
> 調査結果の要点を 1〜3 文で。忙しい読み手はここだけ読む。

## 調査結果の詳細

### 観点 1

データや根拠を示しながら説明。

### 観点 2

比較や分析結果をテーブルで整理。

## 結論と次のアクション

- **結論**: 調査結果から導かれる判断
- **次のアクション**: 具体的に誰が何をすべきか

## 参考リンク

| 情報源 | URL |
|---|---|
| 名称 | リンク |
```

## レポート種別ごとの構成パターン

### 技術調査レポート

技術選定や方式比較を行う場合の構成。

- **背景**: 技術課題・要件を整理
- **選択肢の全体像**: 比較対象を並べ、推奨・非推奨を明示
- **各選択肢の詳細**: メリット・デメリット・構成図を含む
- **結論と次のアクション**: 推奨案と理由、具体的な次のステップ

選択肢の比較では、以下のように推奨度を明示する:

```markdown
## 選択肢1: ○○方式 — 推奨

（推奨理由と詳細）

## 選択肢2: △△方式 — 非推奨

（非推奨理由と詳細）
```

### データ分析レポート

数値データの異常・傾向を調査する場合の構成。

- **問題の概要**: 何が起きたか — 期待値と実際の差を数値で示す
- **調査結果の概要**: 原因を端的に述べ、根拠データを提示
- **結論と対応案**: 対応内容・対象者・優先度をテーブルで整理
- **調査の詳細**: 詳細なデータテーブルや日別分析（必要に応じて折りたたみ）

数値の強調パターン:

```markdown
| 項目 | 欠損あり（3/2） | 正常（3/3） |
|---|---:|---:|
| レコード数 | **21件**（74件欠損） | **95件**（全件あり） |
| 就業人数 | **0名** | **38名** |
| 実労働時間 | **0.0h** | **284.67h** |
```

### ログ分析レポート

システムログのエラー分析を行う場合の構成。

- **サマリー**: エラー種類数・重大度・最終結果を一覧
- **エラーごとの詳細**: 発生時刻・発生箇所・エラーメッセージ・原因・影響
- **対応状況**: 解決済み / 未解決 / 要対応をステータスで明示

エラー情報はテーブルで整理する:

```markdown
| 項目 | 内容 |
|---|---|
| 発生時刻 | 2026/02/13 09:27:58 |
| 発生箇所 | `import_companyData.vbs`（ログ111行目） |
| エラーメッセージ | `424 オブジェクトがありません。` |
```

## 図解の活用

レポートでは構成図やフロー図が特に有効。図の種類の選び方は `diagram-patterns.md` を参照する。

| 場面 | 推奨する図 |
|---|---|
| システム構成の比較 | `flowchart` + `subgraph` で選択肢を並列表示 |
| 通信・処理の流れ | `sequenceDiagram` で時系列を表現 |
| データの差分・変化 | テーブルで数値比較（図より表が適切） |

## レポート固有のルール

- **結論を先に書く** — 背景→詳細→結論の順ではなく、背景→結論→詳細の順にする。読み手は結論を知りたい
- **数値には必ず単位をつける** — `284.67` ではなく `284.67h` や `284.67時間`
- **比較には必ずテーブルを使う** — 箇条書きで比較しない
- **推奨・非推奨は明示する** — 曖昧にせず判断を示す
- **次のアクションは具体的に** — 「検討する」ではなく「○○に確認する」
- **参考リンクはテーブルで整理する** — 情報源名とURLのペア

# スキル作成ガイド

AI エージェントのスキル（SKILL.md）を新規作成・更新する際のルールを定義する。

## テンプレート

````markdown
---
name: <skill-name>
description: <用途 + ユーザーが言いそうなフレーズ 2〜3 個>
---

# <スキルタイトル>

<スキルの目的を1〜2文で説明>

## 最初にやること / 鉄則（該当スキルのみ）

<冒頭ルーティングまたは常に守るべき原則>

## 前提条件（該当スキルのみ）

<実行に必要な環境・ツール・状態>

## ルール / ガイド

<本文>

## リファレンス（references/ がある場合）

| ファイル | 内容 | いつ読むか |
|---------|------|-----------|
| `references/xxx.md` | ... | ... |

````

## 構成ルール

1. ディレクトリ名は **kebab-case**、`name` と一致させる
2. `description` は日本語 1 行で**用途**と**トリガーフレーズ**を含める（書き方は `references/design-patterns.md` 参照）
3. `disable-model-invocation: true` は**コマンドモード専用スキルのみ**に付ける。通常のスキルは description ベースの自動トリガーを前提とするため付けない
4. 見出しレベルは **H2 → H3 → H4** の順で飛ばさない
5. 見出しに絵文字を使わない — モデルが見出しをパースする際にノイズになり、検索性も落ちるため
6. 注意書きには **GitHub Markdown Alerts**（`> [!WARNING]` 等）を使う — `> ⚠️` 形式はレンダラーによって表示が崩れ、意味が伝わらない場合があるため
7. サンプルには汎用的な名前を使う — 会社名・製品名・社内システム名などの固有名詞を含めない。固有名詞があると、他の場面で使う際に「この例は自分には関係ない」とモデルが誤判断するため（プロジェクト専用スキルを除く）
8. スキルを追加・変更したら **README.md** のスキル一覧・構成図・個別説明も更新する

## 冒頭ルーティング

以下に該当するスキルは冒頭セクション（`## 最初にやること` または `## N つの鉄則`）を置く:

- `scripts/` にスクリプトがあるスキル
- 実行順序（Phase、Step）があるスキル
- サブエージェント委任ルールがあるスキル
- 環境や条件によって最初のアクションが分岐するスキル

**5行以内で「判断→アクション」のみ書く。** モデルは SKILL.md を上から順に読むため、冒頭が長いと本題に入る前にコンテキストを消費する。背景説明は後続セクションに置く。

## 記述の質

スキルの目的はモデルに**意図通りの行動をさせること**であり、体裁を整えることではない。以下の原則を守る:

### 逆ピラミッド構造

SKILL.md の先頭に近いほどモデルの注意が高い。最も重要なルール・判断ロジックを冒頭に、補足・例外を後半に置く。

### ルールには理由を添える

モデルは「〜すること」より「〜の理由で〜する」の方が安定して従う。命令だけのルールは表面的にしか守られず、想定外の場面で破られやすい。

```markdown
<!-- ❌ 理由がない -->
原文テキストを改変しないこと。

<!-- ✅ 理由がある -->
原文テキストを一字一句改変しない — 議事録は発言の正確な記録であり、要約や言い換えは事実の歪曲になるため。
```

### 検証可能なルールを書く

曖昧なルールはモデルが自己判断で拡大解釈する。「高品質な出力を心がける」は検証不能。「全列・全行を省略せず変換する」は検証可能。

### モデルの暴走を予測する

モデルは指示を過剰に解釈しがち。「メールを検索して」と言われると全フォルダを再帰検索し始める、等。スキルには**やるべきこと**だけでなく**やらなくていいこと**の境界も示す。ただし否定形の乱用は避け、境界が必要な場面に限定する。

詳細なパターンと具体例は `references/design-patterns.md` を参照。

## 記述スタイル

### 理由 → 行動の順で書く

ルールの背景や意図を先に示し、その上で行動を指示する。モデルは理由を理解すると、明示されていないエッジケースにも適切に対応できる。

ハードストップ（セキュリティ上の絶対禁止事項、データ損失リスク）に限り、命令形で端的に書いてよい。

### 重複ルールを避ける

1つのルールは1箇所にのみ記述する。重複が必要な場合は参照を書く。

## トークン量と references/ 分離

スキル本文は呼び出し時に全文ロードされるため、簡潔さが重要。

- 目標: **200 行以下**
- 上限: **400 行**
- 超える場合は `references/` に分離し、SKILL.md にリファレンステーブルを置く
- 分離基準・ファイル命名・テーブル書式は `references/design-patterns.md` 参照

## ディレクトリ構成

```
skill-name/
├── SKILL.md           # 必須: スキル定義
├── references/        # 任意: 詳細仕様・例集（SKILL.md から参照）
├── scripts/           # 任意: 独立実行スクリプト（argparse + --help 対応）
└── lib/               # 任意: scripts/ から import する共有モジュール
```

- `scripts/` のスクリプトがある場合、冒頭ルーティングに対応表を置く
- `lib/` は直接実行しない共有コードに使う
- 再利用するまとまったコード（20行以上目安）は `scripts/` に配置する。短い使用例（数行のコマンドやスニペット）は SKILL.md にインラインで書いてよい

## スキルのスコープ

### スキルにすべきか、CLAUDE.md に書くべきか

- **全セッションで常に適用する短いルール**（5行以下）→ CLAUDE.md
- **特定のトリガーで発動し、まとまった手順・判断ロジックがある** → スキル

CLAUDE.md は常にコンテキストに載るため、長いルールを書くとすべての会話でトークンを消費する。スキルは必要なときだけロードされるため、コンテキスト効率が良い。

### 分割・統合の判断

- 1つのスキルが 400 行を超えたら分割を検討する
- 2つのスキルが常にセットで使われるなら統合を検討する
- 同じ領域で用途が異なるスキル（例: 設定 vs 操作）は分けたまま、description で差別化する

## スキル間参照

別スキルのルールを参照する場合:

- 同一スキル内: 「§X 参照」
- 別スキル: 「`<skill-name>` の §X を参照」

機能が重複するスキルが既にある場合は、新規作成せず既存スキルを参照する。

## チェックリスト

### 構成

- [ ] ディレクトリ名が kebab-case、`name` と一致
- [ ] `description` にトリガーフレーズが 2〜3 個含まれている
- [ ] `disable-model-invocation: true` がコマンドモード専用スキル以外に付いていない
- [ ] 冒頭ルーティングがある（該当スキルのみ）
- [ ] 前提条件が明記されている（該当スキルのみ）
- [ ] 見出しレベルが飛んでいない・絵文字が含まれていない
- [ ] 注意書きが GitHub Markdown Alerts を使っている
- [ ] 本文が 400 行以下（超える場合は `references/` 分離済み）
- [ ] `references/` のリファレンステーブルに「いつ読むか」列がある
- [ ] README.md のスキル一覧・構成図・個別説明が更新されている

### 内容の質

- [ ] 最も重要なルールが SKILL.md の前半にある（逆ピラミッド）
- [ ] 主要なルールに「なぜそうするか」の理由が添えられている
- [ ] ルールが検証可能（「高品質にする」のような曖昧な指示がない）
- [ ] 具体例がある（ルールだけでは解釈がブレる）
- [ ] やるべきことだけでなく、境界（やらなくていいこと）が必要な箇所に示されている

### 検証

- [ ] 想定するトリガーフレーズでスキルが発動することを確認した
- [ ] スキルの指示に従って実際にタスクを実行し、期待通りの結果が得られた

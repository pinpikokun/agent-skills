# スキル設計パターン

## description の書き方

description はスキルの自動トリガーを決定する最重要フィールド。AI エージェントはセッション開始時にすべてのスキルの description を読み込み、ユーザーの発言に合致するスキルを選択する。

### 構成要素

1. **用途**: スキルが何をするか（1文）
2. **対象範囲**: どういう状況で使うか
3. **トリガーフレーズ**: ユーザーが実際に言いそうな文言を `「〜して」` の形で 2〜3 個

### 良い例

```yaml
# 具体的なトリガーフレーズがある
description: >
  Markdown 議事録・レポートを高品質な HTML に変換。
  参照 HTML のデザインを厳密に踏襲する。
  「議事録を HTML にして」「レポートを HTML 化して」と依頼されたときに使用する。

# 類似スキルとの差別化が明確
description: >
  Outlook メール操作（Python COM）。大量メールの検索・集計・分析、
  添付ファイル一括保存、本文からのデータ抽出など、解析を伴う重い処理向け。
description: >
  Outlook メール操作（PowerShell COM）。メールの検索・作成・送信・返信・転送。
  Windows 標準環境のみで動作する軽量処理向け。
```

### 悪い例

```yaml
# トリガーフレーズがない
description: Git 操作の安全ルール。

# 抽象的すぎる
description: 設定ファイルの管理。

# 用途が不明瞭で類似スキルと区別できない
description: Outlook のメール操作を行うスキル。
```

### 類似スキルがある場合

同じ領域のスキルが複数ある場合（例: outlook-python と outlook-powershell）、description で**用途の違い**を明示する。ユーザーの発言だけでどちらを使うべきか判断できるようにする。

- 処理の重さで分ける: 「大量処理・分析向け」vs「軽量処理向け」
- 技術スタックで分ける: 「Python COM」vs「PowerShell COM」
- ユースケースで分ける: 「設定・トラブルシューティング」vs「チケット CRUD 操作」

## references/ の分離パターン

### SKILL.md に残すもの

- 冒頭ルーティング（判断→アクション）
- 前提条件
- コアルール（常に参照が必要な情報）
- リファレンステーブル

### references/ に出すもの

| 分離パターン | 内容 | ファイル名の例 |
|---|---|---|
| API リファレンス | ツール・クラスの一覧と引数 | `api.md` |
| ワークフロー詳細 | 検索手順、フィルタ条件等 | `search-workflow.md` |
| 出力フォーマット | テーブル形式、レポート形式等 | `output-format.md` |
| エージェント仕様 | サブエージェントの役割・起動方法 | `agents.md` |
| デザイン仕様 | CSS変数、コンポーネント数値 | `design-tokens.md` |
| 具体例集 | コード例、図の書き方例 | `mermaid-examples.md` |
| アンチパターン | 避けるべきパターンと代替 | `anti-patterns.md` |

### ファイル命名

- kebab-case
- 内容が推測できる名前にする（`ref-1.md` のような連番は不可）

### リファレンステーブルの書式

```markdown
| ファイル | 内容 | いつ読むか |
|---------|------|-----------|
| `references/api.md` | OutlookClient API 一覧 | コードを書くとき |
```

**「いつ読むか」列は必須。** この列がないとモデルがすべてのファイルを読み込み、トークンを浪費する。

## サブエージェント委任パターン

大量テキスト（メール本文、チケット説明文等）を扱うスキルでは、読み取り操作をサブエージェントに委任してメインコンテキストの圧迫を防ぐ。

### 基本ルール

| 操作 | 実行場所 | 理由 |
|---|---|---|
| 読み取り（検索・取得） | サブエージェント | 本文が長くコンテキストを圧迫するため |
| 書き込み（作成・更新・送信） | メイン | ユーザー確認が必要なため |

### SKILL.md での書き方

冒頭ルーティングで委任ルールを明示する:

```markdown
## 最初にやること

1. **読み取り操作** → `general-purpose` サブエージェントに委任する
2. **書き込み操作** → メインで直接実行する
```

このパターンが不要なスキル（読み取りデータが小さい、サブエージェントが使えない環境を想定）では省略してよい。

## 記述の質 — 詳細パターン

SKILL.md の「記述の質」セクションの原則を、具体例で補足する。

### 逆ピラミッド構造の実践

```markdown
<!-- ❌ 背景から入る — モデルが本題にたどり着くまでにコンテキストを消費 -->
## 背景
Outlook COM は Windows 環境で...（30行の説明）

## 注意事項
...

## 実際の使い方
ここまで読んでやっと本題

<!-- ✅ 最重要の判断ロジックが先頭 -->
## 最初にやること
1. 検索 → サブエージェントに委任
2. 送信 → メインで実行

## 前提条件
Outlook が起動していること

## 詳細ルール
...（以降は必要に応じて読む情報）
```

### 良いルール vs 悪いルール

| 特徴 | 良いルール | 悪いルール |
|---|---|---|
| 検証可能か | 「全列・全行を省略せず変換する」 | 「高品質な出力を心がける」 |
| 理由があるか | 「議事録は正確性が最重要のため、原文を改変しない」 | 「原文を改変しないこと」 |
| 具体的か | 「`Display()` で下書き表示し、`Send()` はユーザーが求めた場合のみ」 | 「メール送信は慎重に行う」 |
| 境界が明確か | 「受信トレイと送信済みフォルダのみ検索する」 | 「メールを検索する」 |

### モデルの暴走パターンと対策

モデルには予測可能な傾向がある。スキルを書くときにこれらを意識すると、意図しない動作を防げる。

| 傾向 | 例 | 対策 |
|---|---|---|
| **過剰実行** | 「検索」と言われて全フォルダを再帰検索 | 検索対象のデフォルト範囲を明示する |
| **先回り** | 頼まれていないのに git commit を提案 | 「ユーザーの指示を待つ」を明記する |
| **過剰丁寧** | 3行で済む回答を20行で説明 | 出力フォーマットをテーブルや箇条書きで指定する |
| **コンテキスト汚染** | 大量データをメインに取り込んで後続が破綻 | サブエージェント委任パターンを使う |
| **拡大解釈** | 「改善して」を全体リファクタリングと解釈 | スキルの適用範囲を具体的に限定する |

対策の基本方針: **禁止するのではなく、デフォルトの行動を定義する。** 「〜してはいけない」より「デフォルトでは〜する。ユーザーが求めた場合のみ〜する」の方が、モデルが判断しやすい。

## 検証ガイド

スキルを書いたら、以下の手順で動作を確認する。

### トリガー確認

1. description に含めたトリガーフレーズで実際に呼びかけてみる
2. フレーズを少し変えても発動するか確認する（「HTML にして」→「HTML 化して」→「HTML に変換して」）
3. **似ているが別のスキルが適切な依頼**で誤発動しないか確認する

### 動作確認

1. スキルの主要なユースケースを1つ実行する
2. 冒頭ルーティングの判断分岐が正しく動くか確認する
3. 出力が期待通りか確認する（省略・改変・過剰追加がないか）

### よくある問題

| 症状 | 原因 | 対処 |
|---|---|---|
| スキルが発動しない | description のフレーズがユーザーの言い方と乖離 | トリガーフレーズを追加・修正 |
| 間違ったスキルが発動する | description が類似スキルと差別化できていない | 用途の違いを description に明記 |
| 手順の一部が飛ばされる | 冒頭ルーティングがなく、重要な手順が後半に埋もれている | 逆ピラミッド構造に再構成 |
| 想定外の動作をする | ルールが曖昧、または境界が不明確 | 検証可能なルールに書き直し、デフォルト動作を明記 |
